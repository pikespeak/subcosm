# Phase 2.1: Code Review Report

**Reviewed:** 2026-06-20T10:32:00Z
**Depth:** standard
**Files Reviewed:** 7

## Summary

Phase 2.1 (VIS-DEPTH / VIS-ANIM) is in strong shape on the project's highest-value
invariants: determinism is preserved (no `Math.random`, no Devvit imports in
`src/engine/`; the new radius/weight code consumes ZERO `rng()` calls and reads only
`idx`/`N`/already-derived `day` scalars — verified against the frozen
`elementsBaseline` byte test), the Zod single-source-of-truth rule holds (`Shell.weight`
is a schema field with `z.infer`, no hand-written interface, no `as` casts), contract
separation holds (synthesis stays Phaser-free; paint and `ignite.ts` read only Scene
values, never a raw DayVector), and the no-strobe bound in `ignite.ts` is genuinely
proven (analytic `[SHIMMER_BASE − AMPLITUDE_MAX, SHIMMER_BASE + AMPLITUDE_MAX]` =
`[0.60, 0.96]`, swept across the full input domain in the test). The new shimmer math
sits correctly below the `if (!this.animate ...) return;` gate, and the frozen-shell
signature is baked once.

However, there is one BLOCKER: the new `shellRadius` falloff produces **negative radii
at the explicitly-stated production scale (N≈30)**. The inner shells cross through and
collide with the genesis core, and the oldest shell vanishes entirely — the exact
"central blob / illegible deep rings" failure D-01 exists to prevent. The test suite
only exercises the N=3 fixture, so it passes green while the production geometry is
broken. The remaining findings are stale/contradictory documentation on the
reduced-motion accessibility path and a couple of robustness gaps.

## Critical Issues

### CR-01: `shellRadius` produces negative radii at the production target N≈30 — inner shells collide with the core and the oldest shell disappears

**File:** `src/engine/synthesis.ts:103-114` (consumed at `:224`)
**Issue:**
The clamped-min-gap falloff enforces a *minimum* gap inward (`Math.min(prevRadius - minGapFor(n), blended)`) but never clamps the radius to stay positive / above the
core. `minGapFor(n) = 0.9/n`, so over N rings the cumulative forced descent is up to
`0.9*(n-1)/n ≈ 0.9`, starting from a frontier radius of only ~1.0. Past idx≈28 the
radius goes negative.

Simulated (pure reimplementation of the shipped function):
```
N=30  idx=27 r=0.0434  idx=28 r=0.0134  idx=29 r=-0.0166   ← negative
N=50  last r=-0.1071
N=80  last r=-0.1735
```
The genesis core sits at `radius: 0.06` (`synthesis.ts:254`). So at N=30:
- idx 28-29 land **at or below the core** → the "every ring distinguishable *to the
  core*, no central blob" guarantee (D-01) is violated precisely where the code comment
  claims it holds ("the clamped-min-gap floor scales with N so the guarantee holds [at
  N≈30]", `synthesis.test.ts:69`).
- In `paint.ts`, a negative `shellR` (`shell.radius * rMax`) hits `if (shellR < 1.5)
  continue` (`paint.ts:279`) and the guide-ring `if (r >= 2)` guard (`paint.ts:272`), so
  the **oldest shell is silently dropped entirely** — visible data loss (the oldest
  day's ring disappears).
- The near-zero shells (idx 27-28) that survive map to sub-pixel radii stacked on the
  core, i.e. the central blob D-01 is meant to prevent.

The N=3 fixture never reaches this (last radius 0.40), so `synthesis.test.ts` is green
and the defect is uncaught. The test asserting D-01 (`:70-83`) only runs at N=3.

**Fix:** Clamp the falloff so the innermost ring stays a bounded margin above the core,
and add a test at production scale. One approach — reserve headroom above the core and
distribute the rings within `[coreMargin, 1]` so the gap shrinks gracefully instead of
going negative:
```ts
const CORE_MARGIN = 0.12; // innermost ring never closer than this to center (core=0.06)

export function shellRadius(idx: number, n: number, prevRadius: number | null): number {
  const expPart = Math.pow(RADIUS_EXP_BASE, idx);
  const linPart = n > 0 ? 1 - idx / n : 1;
  const blended = expPart * RADIUS_EXP_WEIGHT + linPart * RADIUS_LIN_WEIGHT;
  // Re-map [0,1] falloff into [CORE_MARGIN, 1] so deep rings never reach the core.
  const mapped = CORE_MARGIN + (1 - CORE_MARGIN) * blended;
  if (idx === 0 || prevRadius === null) return mapped;
  // Keep the min-gap pull, but floor it above the core so radius stays positive.
  return Math.max(CORE_MARGIN, Math.min(prevRadius - minGapFor(n), mapped));
}
```
Then add a regression test at N≈30 asserting `radii[n-1] > core.radius` and all radii
strictly positive and strictly decreasing — the current suite cannot catch this class
of bug because it only synthesizes the 3-day fixture.

## Warnings

### WR-01: Stale comments claim the reduced-motion rest frame uses `pulse/twinkle = 1`, but it actually uses `0.78 / 0.85`

**File:** `src/client/cosmos/CosmosScene.ts:203-205` and `src/client/cosmos/paint.ts:339-343`
**Issue:**
`renderFrontier`'s docstring states "`pulse`/`twinkle` are 1 for the static
reduced-motion frame (no animation)" and `drawIgniteRing`'s docstring states "under
reduced-motion it is called once with pulse=1 (fully on, no strobe)". After the
VIS-ANIM rework the actual rest values are `IGNITE_REST = { pulse: SHIMMER_BASE = 0.78,
twinkle: 0.85 }` (`ignite.ts:112`), passed at `CosmosScene.ts:142,169`. These are
left-over comments from the legacy `[0,1]` pulse where rest was 1. The reduced-motion /
post-nudge static frame is a load-bearing accessibility path (PNT-04); comments that
misstate its value by 22% will mislead the next maintainer reasoning about whether the
static frame matches the animated baseline.
**Fix:** Update both docstrings to reference `IGNITE_REST` / `SHIMMER_BASE`, e.g. "drawn
once at the shimmer rest value `SHIMMER_BASE` (0.78) — the sine's zero-crossing, so the
static frame and the animated baseline agree (no strobe)."

### WR-02: `this.animate` derived from `motion.frontierOnly` inverts the flag's meaning — a full-motion style gets NO animation

**File:** `src/client/cosmos/CosmosScene.ts:83,106`
**Issue:**
`this.animate = this.style.motion.frontierOnly && !prefersReducedMotion();`. The schema
field `frontierOnly` means "only the frontier animates (vs everything animating)" — it
is a *scope* flag, not an on/off switch. Used as the animation gate, a style that
intends full animation (`frontierOnly: false`) silently disables the frontier pulse
entirely. The docstring at `:82` even says "the style says the frontier animates", which
`frontierOnly` does not express. Today every preset sets `frontierOnly: true` so it
works, but this is a latent trap the moment a non-frontier-only style ships.
**Fix:** Gate animation on motion being enabled, and treat `frontierOnly` as the scope
hint it is:
```ts
// animate whenever motion is allowed; frontierOnly stays a scope hint (we only ever
// animate the frontier regardless, per the perf model).
this.animate = this.style.motion.speed > 0 && !prefersReducedMotion();
```
(or introduce an explicit `motion.enabled`/`reducedFallback` field if speed=0 should
still draw a static-only style). At minimum, fix the docstring to state the real
semantics.

### WR-03: `update(time)` reads the Phaser absolute clock, but `igniteParams` was bound/verified against an elapsed-ms assumption

**File:** `src/client/cosmos/CosmosScene.ts:178,192-197` (param `time`); `src/client/cosmos/ignite.ts:77,95`
**Issue:**
`override update(time: number)` passes Phaser's `time` straight into
`igniteParams(..., time, ...)`, where it is used as `Math.sin(time * tempo)`. Phaser's
`update(time, delta)` `time` is the absolute timestamp since game boot (can be large),
not elapsed-since-create. The no-strobe bound holds for any `time` (the sweep covers
`0..60s`), so this is not a correctness blocker for the bound — but the *phase* of the
shimmer depends on absolute boot time, so the pulse will be at an arbitrary phase on
first visible frame and is not reset on resize/reduced-motion re-layout. The `_time`
param in `renderFrontier` (`:206`) is unused, suggesting the time-handling was
refactored mid-stream. Worth confirming the intended clock.
**Fix:** If a stable phase from scene-create is intended, store `this.startTime` in
`create()` and pass `time - this.startTime`. If absolute time is acceptable, drop the
unused `_time` parameter from `renderFrontier` and add a one-line comment that absolute
clock phase is intentional.

### WR-04: D-01 test hardcodes `MIN_GAP = minGapFor(3)` and only ever runs at N=3 — the gap/no-blob guarantee is unverified at any other shell count

**File:** `src/engine/synthesis.test.ts:65-83`; `src/engine/synthesis.ts:91`
**Issue:**
`MIN_GAP` is a fixed `minGapFor(3)` and every depth-geometry test synthesizes only the
3-day `fixtureDays`. The "guaranteed minimum gap" and "strictly decreasing" assertions
therefore prove nothing about the production N≈30 path — which is exactly where CR-01
breaks. The test comment even advertises N≈30 as the target ("the clamped-min-gap floor
scales with N so the guarantee holds there") without ever exercising it.
**Fix:** Add a parametrized test that builds an N=30 (and N=50) synthetic
`DayVector[]` and asserts: all radii strictly decreasing, all radii > `core.radius`, all
radii > 0, and each adjacent gap ≥ `minGapFor(N) - 1e-9`. This both documents the
contract at scale and would have caught CR-01.

## Info

### IN-01: `shellWeight` accent term silently clips above weight 0.75 (lost resolution, not a bug)

**File:** `src/engine/synthesis.ts:134-135`
**Issue:** `weighted = ageFade*0.7 + standout*0.4` can reach `1.1` for a fresh
high-standout shell, then `clamp01` flattens everything ≥ ~0.857 standout to exactly 1.
So the "standout accent" loses all resolution at the high end — two very different
high-conflict frontier days render at identical weight 1.0. Within spec (bounded [0,1],
floor respected) but the accent's top quartile is dead.
**Fix:** If full-range accent is desired, normalize the weights (e.g. `0.6*ageFade +
0.4*standout`, max = 1.0) so no clipping occurs.

### IN-02: `MIN_GAP` exported constant is misleadingly named (it is the N=3 value, not "the" min gap)

**File:** `src/engine/synthesis.ts:89-91`
**Issue:** `MIN_GAP` reads like a global invariant but is only `minGapFor(3)`. The
runtime floor is per-N. A reader importing `MIN_GAP` elsewhere would get a value that is
wrong for any N≠3.
**Fix:** Rename to `MIN_GAP_AT_N3` (or inline `minGapFor(3)` in the test and drop the
export), making the fixture-specificity explicit.

### IN-03: Genesis-core fallback can diverge from the genesis shell's `weight` (cosmetic)

**File:** `src/engine/synthesis.ts:251-257` vs `:231`
**Issue:** The genesis *shell* gets a `weight` via `shellWeight`, but the genesis
*core* node has no weight concept and derives energy/hue independently
(`genesis.contributors / 100`, `hueHint`). Not a defect, but if `weight` is later meant
to govern core legibility too, the two paths will need reconciling. Noting for the
forward-constraint (embeddable Scene) tracking only.
**Fix:** None required now; flag if core legibility ever becomes weight-driven.
