# Phase 2.1: Visual Depth & Animation Polish - Research

**Researched:** 2026-06-19
**Domain:** Deterministic generative-graphics geometry (radial spacing) + data-driven Phaser/WebGL frontier animation, under a strict synthesis/paint contract split
**Confidence:** HIGH (codebase fully grounded; the only external dependency — Phaser 4.2 DynamicTexture bake — verified via Context7 and matches existing `bake.ts`)

## Summary

This phase is entirely a **tuning pass inside an existing, well-factored pipeline** — no new packages, no external services, no Reddit code. Every change lands in code that already exists: the `radius` value emitted by `src/engine/synthesis.ts` (VIS-DEPTH), the per-frame `pulse`/`twinkle` math in `src/client/cosmos/CosmosScene.update()` (VIS-ANIM), the `drawIgniteRing()` seam in `paint.ts`, and the `bakeShell()` DynamicTexture path (D-05). The architecture already enforces the two hard contracts (synthesis is Phaser-free, paint never reads raw DayVector), so the research question is *where each change must live to respect those seams*, plus concrete formulas the planner can pick from.

The decomposition is clean: **D-01 (radius geometry) and D-02 (per-shell brightness/size) are synthesis-side** — they change values on the `Scene`/`Shell` contract and therefore re-baseline the golden snapshot in `synthesis.test.ts`. **D-03/D-04 (live ignite/shimmer) are paint-side** — they change `CosmosScene.update()` reading `shell.energy` + `shell.meta.conflict`, which are *already on the Scene shell*, so no contract change and no re-baseline. **D-05 (frozen-shell character) is also paint-side** — it injects per-shell variance into the existing `drawShell → bakeShell` path at zero per-frame cost. The one contract decision the planner must make explicitly: `momentum` is **not** on the Shell today (only on DayVector), so any momentum-driven drift requires either adding a field to `ShellSchema` (→ contract change + re-baseline) or deriving drift from a metric already on the shell.

**Primary recommendation:** Split the phase into two waves — **Wave A: synthesis geometry + per-shell weight (D-01, D-02)** which re-baselines the snapshot once; **Wave B: paint animation (D-03, D-04, D-05)** which touches no contract and no snapshot. Use a **clamped-minimum-gap hybrid** for the radius (perspective falloff toward the core but a hard floor on inter-ring spacing), keep brightness/size as **synthesis-emitted per-shell scalars with a legibility floor**, and implement the ignite as **gentle shimmer baseline + bounded conflict/energy modulation** with a hard cap that mathematically cannot strobe.

### Pattern 1: Radius falloff with a guaranteed minimum gap (D-01)
**What:** Replace `radius: Math.pow(0.85, idx)` with a mapping that still compresses toward the core (perspective/depth feel) but guarantees every adjacent ring is separated by at least a floor gap, so ~30 rings stay individually distinguishable.

**When to use:** This is the locked D-01 behavior. The exact curve is Claude's discretion.

**Candidate formulas (all deterministic, no RNG, no `Math.random()`):**

The `radius` is normalized to roughly `0..1` (paint multiplies by `rMax`). Let `idx` = 0 (frontier) .. `N-1` (oldest), `N≈30`.

- **(A) Clamped-minimum-gap hybrid (RECOMMENDED).** Keep an exponential shape but blend toward linear with a hard floor on the per-ring step:
  ```ts
  // r decreases as idx grows; frontier (idx 0) is largest.
  const N = days.length;
  const minGap = 0.9 / N;          // floor: even spacing would give 1/N; take ~0.9 of it
  const expPart = Math.pow(0.92, idx);            // gentler than 0.85 → less crushing
  const linPart = 1 - idx / N;                    // pure even spacing
  const blended = expPart * 0.45 + linPart * 0.55;
  // enforce a monotonic minimum gap from the previous ring:
  radius = (idx === 0) ? blended : Math.min(prevRadius - minGap, blended);
  ```
  Tradeoff vs mock: deep rings no longer collapse into a blob; the extreme "infinity tunnel" of `0.85^i` is softened. Reads closer to a legible onion than a vanishing point. **This best satisfies success criterion #1.**

- **(B) √ (square-root) spacing.** `radius = 1 - Math.sqrt(idx / N)` — naturally large gaps near the frontier, gaps shrink toward core but never to zero before a clamp. Simpler, one-line, but less of a perspective feel than (A).

- **(C) Logarithmic-with-floor.** `radius = 1 - Math.log(1 + idx) / Math.log(1 + N)`, then clamp inter-ring gap to `minGap`. Strong depth compression early, very even late — risk: still crowds the deep core unless the clamp does the heavy lifting.

- **(D) Pure arithmetic (rejected by D-01).** `radius = 1 - idx/N`. Loses depth/infinity feel; D-01 explicitly rejects equal-spacing alone. Useful only as the `linPart` term inside (A).

**Recommendation:** **(A)**. It is the literal embodiment of "perspective falloff WITH a guaranteed minimum gap." Expose `minGap` constant and the blend weight as local constants (or, if a preset should vary depth feel, as a Genome knob — but that is scope creep; keep them local constants for this phase). `[ASSUMED: formula choice — verify against the mock side-by-side during execution per success criterion #1]`

**Determinism note:** all four are pure functions of `idx`/`N` — no RNG consumption added, so the per-day `mulberry32` stream is untouched and *element* positions stay byte-identical. Only the `radius` scalar changes → the snapshot re-baseline is limited to that field.

### Pattern 2: Per-shell brightness/size weight with a legibility floor (D-02)
**What:** Each shell fades with age but standout days (high conflict/energy) keep accents; nothing fades below a legibility floor.

**Where it lives (the contract decision):** Two valid placements —
- **(2a) Synthesis emits a per-shell `weight` scalar (RECOMMENDED).** Add `Shell.weight: z.number().min(0).max(1)` (or reuse a floor-clamped composite). Synthesis computes it from age + the day's `conflict`/`energy`:
  ```ts
  const ageFade = lerp(1, 0.45, idx / N);                 // older = dimmer
  const standout = Math.max(day.conflict, /*energyProxy*/ ...);
  const weight = Math.max(LEGIBILITY_FLOOR, ageFade + standout * 0.4); // floor ~0.35
  ```
  Paint then multiplies `weight` into the per-glow alpha and star size in `drawShell` (`energyAlpha`, `sz`). **Pro:** "which days matter" is *data about the day*, belongs in synthesis; testable in `synthesis.test.ts`. **Con:** ShellSchema contract change → snapshot re-baseline (already user-approved per CONTEXT integration note).
- **(2b) Paint-side mapping only.** Synthesis emits nothing new; `drawShell` derives age from `idx` and standout from `shell.meta.conflict` (already present) + mean `element.energy`. **Pro:** zero contract change, no re-baseline. **Con:** paint now encodes the *policy* "old fades, conflict stands out," which is arguably synthesis's job; energy isn't a clean per-shell scalar (only per-element).

**Recommendation:** **(2a)** — it keeps the synthesis/paint split honest (paint maps a number to pixels; it does not decide which days are notable) and the snapshot re-baseline is already approved. If the planner wants to avoid *any* contract churn, (2b) is acceptable since `shell.meta.conflict` is already on the shell and `idx` gives age — but document that paint is then carrying a data-policy decision. **Note `Element.energy` already exists** and `drawShell` already does `fillAlpha * (0.6 + el.energy * 0.4)` — D-02 layers a *shell-level* weight on top of that *element-level* energy.

### Pattern 3: Gentle shimmer + bounded data-driven ignite (D-03/D-04)
**What:** Replace the harsh `pulse = 0.55 + 0.45*sin(time*0.0022*speed)` (range 0.10→1.0, reads as on/off blink) with a soft shimmer baseline plus conflict/energy modulation that mathematically cannot strobe.

**Locked mapping (D-03):** `conflict` → pulse hardness/amplitude; `energy` → pulse tempo/speed; `momentum` → optional slow drift.

**The clean seam:** `CosmosScene.update(time)` computes `pulse` and passes it to `drawIgniteRing(...)`. The frontier shell is `this.cosmos.shells[0]`; its metrics are `shells[0].meta.conflict` (0..1, already on shell) and a per-shell energy proxy (mean of `shells[0].elements[].energy`, or — cleaner — have synthesis emit `Shell.energy` like the core has). **No new draw call, no extra per-frame object** — just better math feeding the existing single `strokeCircle`.

**Concrete bounded formula (cannot strobe):**
```ts
const frontier = this.cosmos.shells[0];
const conflict = frontier?.meta.conflict ?? 0;          // 0..1
const energy   = frontierEnergy(frontier);              // 0..1 (mean element energy or Shell.energy)
const speed    = this.style.motion.speed;

// energy → tempo: faster shimmer when the day is energetic
const tempo = 0.0016 * speed * (0.6 + energy * 0.9);    // never 0

// conflict → amplitude/hardness: calm day barely breathes, conflict day pulses harder
const SHIMMER_BASE = 0.78;                              // high rest value = no "off" state
const AMPLITUDE_MAX = 0.18;                             // HARD CAP: swing ≤ ±0.18
const amp = AMPLITUDE_MAX * (0.25 + conflict * 0.75);   // 0.045 (calm) .. 0.18 (max conflict)

// hardness: conflict sharpens the sine toward a harder edge WITHOUT clipping to on/off
const raw = Math.sin(time * tempo);
const shaped = Math.sign(raw) * Math.pow(Math.abs(raw), 1 - conflict * 0.4); // 1.0→0.6 exponent
const pulse = SHIMMER_BASE + amp * shaped;              // mathematically in [0.60, 0.96]
```
**Why it can't strobe:** `pulse ∈ [SHIMMER_BASE - AMPLITUDE_MAX, SHIMMER_BASE + AMPLITUDE_MAX] = [0.60, 0.96]`. The minimum is far from 0, so the ring never blinks off; the cap bounds the swing. Contrast with today's `[0.10, 1.0]`. The `pow`-shaping gives conflict a *harder/sharper* pulse edge without ever reaching an on/off square wave. `[ASSUMED: constants — tune side-by-side per success criterion #2]`

**Twinkle (D-04):** same treatment for the star twinkle (currently `0.72 + 0.28*sin(time*0.004*speed)`). Raise the base, cap the swing, optionally tie tempo to energy. Keep it subtle — twinkle is secondary to the ignite ring.

**Momentum drift (optional, D-03):** `momentum` is **NOT on the Shell** (only DayVector). Two options:
- Skip drift this phase (it's explicitly optional) — simplest, no contract change.
- Add `Shell.momentum: z.number().min(-1).max(1)` → re-baseline, then use it for a slow phase offset: `time*tempo + momentum*driftConst`. Only do this if the planner wants the extra differentiation; it doubles the re-baseline surface.
**Recommendation:** defer momentum drift unless cheap; conflict+energy already satisfy success criterion #2.

### Pattern 4: Baked per-shell signature, zero per-frame cost (D-05)
**What:** A frozen high-conflict day looks visibly different from a calm frozen day even though neither animates. Bake the difference into the shell's DynamicTexture once.

**How the existing bake path works (verified):**
`paintScene` → for each frozen `shells[1..]` → `drawShell(...)` creates per-glow `Image`s with per-object tint/alpha/size → `bakeShell(scene, 'shell-N', objs, w, h)` calls `dt.draw(objs)` which composites them into one DynamicTexture (each Image keeps its own `BlendModes.ADD`), destroys the source objects, and returns one baked `Image`. `[VERIFIED: src/client/cosmos/bake.ts; CITED: phaser DynamicTexture.draw docs]`

**Where to inject variance:** inside `drawShell`, **before** the bake, modulate the per-glow objects by the shell's static metrics (`shell.meta.conflict`, the D-02 `weight`, mean element energy). Because variance is applied to each glow object's own alpha/tint/scale/count, it bakes in for free. Concrete signatures:
- **High conflict:** more redshifted glows (the `el.conflict` redshift flag already exists), tighter angular clumping, a sharper/denser nebula → "turbulent" frozen look.
- **High energy:** brighter core glow, larger `nbig` clusters, more nebula clouds (`nClouds` already scales with element count).
- **Calm:** smoother, dimmer, rounder distribution.

**Critical constraint (from Phaser docs):** `dt.draw()`'s `alpha`/`tint` args **only apply to texture-key strings, not game objects**. So you cannot pass per-shell alpha to `draw([...objects])` and have it work. You must set the variance on each `addGlow`/`addFacetStar` call's own `alpha`/`color`/`radius` arguments (which `drawShell` already controls) **before** baking. `[CITED: phaser DynamicTexture skill docs]`

**Determinism:** if any per-shell variance uses randomness it MUST use the seeded `mulberry32(day.seed)` — but note: D-05 variance computed in *paint* from already-deterministic Scene metrics needs no RNG at all (it's a pure function of `shell.meta.conflict` etc.). Prefer pure-metric variance to avoid any RNG in paint. Synthesis already owns all seeded randomness.

### Anti-Patterns to Avoid
- **Animating frozen shells.** D-05 gives frozen shells *character* (baked), never *motion*. Re-drawing `shells[1..]` per frame violates the 60fps "only the frontier re-renders" rule (PNT-03). Keep all per-frame work in `renderFrontier`.
- **Reading raw DayVector in paint.** Paint must read only `Scene` + `StyleTemplate`. For D-03 use `shell.meta.conflict` / a `Shell.energy` scalar — never reach back to `day.conflict`. (The seam is already correct; don't break it.)
- **Adding RNG to paint.** Any per-shell/per-element variance that needs randomness belongs in synthesis (seeded). Paint variance must be a pure function of Scene values.
- **`Math.random()` anywhere in `src/engine/`.** ESLint-banned (ENG-03). The D-01/D-02 formulas above add zero RNG calls.
- **Strobing.** Any ignite formula whose `pulse` can approach 0 (like today's `0.10`) reads as a blink. Always keep a high shimmer base + capped amplitude (Pattern 3).
- **Changing RNG consumption order in synthesis.** If D-02 adds `rng()` calls inside the per-day loop, it shifts every downstream element's stream → a *total* element re-baseline, not just `radius`. Compute D-02 weight from already-derived day metrics (no new `rng()` calls) to keep the re-baseline surface to just `radius` + the new `weight` field.

## Common Pitfalls

### Pitfall 1: Re-baseline blast radius is larger than intended
**What goes wrong:** A D-02 change that adds `rng()` calls inside the per-day synthesis loop shifts the mulberry32 stream, so *every element's* angle/brightness/redshift changes — turning a one-field (`radius`) re-baseline into a full-Scene re-baseline, and making it hard to verify the change was correct.
**Why it happens:** `mulberry32` is a single sequential stream per day; inserting a consumer anywhere reorders all downstream draws.
**How to avoid:** Compute D-01 `radius` and D-02 `weight` from `idx`/`N`/already-derived day metrics — **no new `rng()` calls**. Then the diff is provably limited to the `radius` field (+ the new `weight` field). Assert this in the test: the *element arrays* should be byte-identical to the pre-change baseline; only `radius`/`weight` differ.
**Warning signs:** the snapshot diff shows changed `angle`/`energy`/`conflict` on elements you didn't mean to touch.

### Pitfall 2: `DynamicTexture.draw` alpha/tint silently ignored on game objects (D-05)
**What goes wrong:** Planner tries to apply per-shell brightness by passing `alpha`/`tint` to `dt.draw(objects, x, y, alpha, tint)`; nothing changes because those args only affect texture-*key* draws, not game objects.
**Why it happens:** Phaser API asymmetry, documented but easy to miss. `[CITED: phaser DynamicTexture skill docs]`
**How to avoid:** Set the variance on each glow object's own `setAlpha`/`setTint`/`setDisplaySize` (i.e. on the `addGlow`/`addFacetStar` arguments in `drawShell`) **before** baking.
**Warning signs:** all frozen shells look identical despite different metrics.

### Pitfall 3: New animation leaks past the reduced-motion gate
**What goes wrong:** A shimmer added directly in `update()` runs even under `prefers-reduced-motion`, producing motion the user asked to suppress (breaks PNT-04 / success criterion #5).
**Why it happens:** Forgetting that `update()` early-returns on `!this.animate`, but `layout()`/`renderFrontier(…, pulse=1)` still draws one static frame.
**How to avoid:** Keep ALL new pulse/shimmer math inside the `if (!this.animate || !this.paintResult) return;` guard in `update()`. The static frame must call the ignite/twinkle with their *rest* values (high shimmer base, no swing). Verify the reduced-motion frame is a single non-strobe render.
**Warning signs:** the canvas animates with reduced motion enabled.

### Pitfall 4: Frontier metric proxy is per-element, not per-shell
**What goes wrong:** D-03 wants "the frontier's energy," but `energy` lives per-*element* on the Scene, not per-shell — averaging it every frame in `update()` is wasteful and noisy.
**Why it happens:** `Shell` has `meta.conflict` but no `energy` scalar (the core does: `core.energy`).
**How to avoid:** Either (a) have synthesis emit `Shell.energy` (a clean per-shell 0..1 scalar — small contract add, re-baseline) and read it directly, or (b) compute the mean **once** when the frontier is set (in `paintScene`/`repaintFrontier`), cache it on the `PaintResult`, and reuse it every frame. Don't recompute per frame.
**Warning signs:** profiler shows array reduction in the hot `update()` path.

### Pitfall 5: Depth feel lost / mock parity drift
**What goes wrong:** Over-correcting D-01 into near-even spacing kills the "universe receding to infinity" feel that the mock's `0.85^i` gives; the side-by-side (success criterion #1 and the visual checkpoints) then reads worse, not better.
**Why it happens:** A flat minimum gap with no exponential component looks like flat concentric circles.
**How to avoid:** Use the hybrid (Pattern 1A) so a perspective component survives; tune the blend weight against the mock at ~30 rings. Judge frontier→day-1 scrub every iteration (the CONTEXT "specifics" mandate this).
**Warning signs:** the cosmos looks like a dartboard, not a depth tunnel.

### Drawing the data-driven ignite (the changed `update()` core)
```ts
// Source: derived from src/client/cosmos/CosmosScene.ts:149-160 (existing seam)
override update(time: number): void {
  this.controller?.update();
  if (!this.animate || !this.paintResult) return;       // reduced-motion gate (PNT-04)

  const frontier = this.cosmos.shells[0];
  const conflict = frontier?.meta.conflict ?? 0;        // already on Scene shell
  const energy   = this.frontierEnergy;                 // cached scalar (Pitfall 4)
  const speed    = this.style.motion.speed;

  const tempo = 0.0016 * speed * (0.6 + energy * 0.9);
  const amp   = 0.18 * (0.25 + conflict * 0.75);
  const raw   = Math.sin(time * tempo);
  const shaped = Math.sign(raw) * Math.pow(Math.abs(raw), 1 - conflict * 0.4);
  const pulse = 0.78 + amp * shaped;                    // ∈ [0.60, 0.96], never strobes

  const twinkle = 0.85 + 0.10 * Math.sin(time * tempo * 2.2);
  this.renderFrontier(time, pulse, twinkle);
}
```
