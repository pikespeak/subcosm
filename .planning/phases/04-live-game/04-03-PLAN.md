<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| realtime channel → client onMessage | App-originated but untrusted-on-the-wire payload; must safeParse + clamp before applying |
| server realtime.send | Only the server may send (the SDK enforces send=server); clients are subscribe-only |
| channel name construction | Must be colon-free (LIVE-01) and identical client/server |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-09 | Tampering | Hostile/garbage realtime message re-synths the frontier to an extreme | mitigate | Client `SteerMsgSchema.safeParse`s every onMessage payload and clamps the aggregate before applying; biases the mean only (I-5); a malformed message is ignored, never thrown on |
| T-04-10 | Spoofing | Client tries to send on the channel as the server | mitigate | The SDK split makes `realtime.send` server-only and `connectRealtime` subscribe-only; clients cannot broadcast |
| T-04-11 | Tampering | Channel-name injection (colon / namespace escape) | mitigate | `steerChannel` whitelists `[A-Za-z0-9_-]` and replaces everything else with `-`; no colons (LIVE-01); built from trusted `context.postId` |
| T-04-12 | Availability | Realtime drops on mobile webview leaving clients out of sync | accept (with fallback) | Realtime is best-effort; the Redis steer hash is the source of truth and the D-03b reload path (plan 02) reconciles on load/reconnect — verified at the blocking on-device checkpoint |
| T-04-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages (RESEARCH Package Legitimacy Audit: zero new deps); realtime ships in the already-installed @devvit/web@0.13.4 |
</threat_model>
