<objective>
Let the mod shape the community at install and make each community freeze at its own local midnight. Declare the install settings (Genome preset + Style + IANA timezone) in `devvit.json` with `validationEndpoint`s returning i18n error keys, read them server-side through `SettingsSchema.parse` (DEV-06); register each installed community + snapshot its config to Redis; and add the hourly UTC cron sweeper that enumerates the registry and fires the plan-03-03 `tick` via `scheduler.runJob` only at each community's local midnight — computed DST-safe from the IANA timezone via native `Intl.DateTimeFormat` with a `hash(subId)%60` jitter (DEV-04 sweeper, D-03).

Purpose: DEV-06 requires the mod to configure the Genome (preset + style) at install via Devvit settings, driving the universe end-to-end with no code changes; DEV-04 requires an hourly UTC sweeper that fires each due community's tick at its local midnight (IANA + jitter, DST-safe). This is the vertical slice that connects "the mod picks Calm + Techno + Europe/Berlin" to "this community freezes at Berlin midnight, not UTC". After this plan, installing the app registers the community with its config and the sweeper will fire its frozen-overnight tick at the right local hour. The tick it fires was built in 03-03; the client that renders the result lands in 03-05.
Output: A `SettingsSchema` boundary, a `schedule.ts` pure local-midnight helper, a `config.ts` read/register service, a `settings.ts` validation router, the `/sweeper` handler in `scheduler.ts`, the install-time registry write in `triggers.ts` (onAppInstall), and the `devvit.json` `settings` + `scheduler.tasks.sweeper` declarations.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mod install settings → server | genome/style/timezone are mod-supplied config crossing into the server |
| Devvit settings store → server read | settings.get values must be validated before driving render/scheduling |
| hourly cron → per-community tick | the sweeper decides which communities freeze; must use the community's own IANA tz, not server-UTC |
| subs:registry growth | the installed-community registry must not grow unbounded or admit a forged sub |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-09 | Tampering | install settings (bad IANA / unknown genome/style) | mitigate | `validationEndpoint` (runtime Intl IANA check) + `SettingsSchema.parse` enum-constrains genome/style; i18n error keys; reject unknown ids (V5) |
| T-03-02 | Spoofing / Info disclosure | sub id source on install/register | mitigate | Derive `sub` from `context.subredditId` server-side in onAppInstall; never register a client-supplied sub (V4) |
| T-03-10 | Tampering (data integrity) | sweeper fires at wrong day boundary | mitigate | Local-midnight gating via native `Intl.DateTimeFormat` per the community's IANA zone (DST-safe), never server-UTC `Date.getHours()`; `hash(subId)%60` jitter spreads load |
| T-03-07 | Tampering (data integrity) | sweeper double-fires / overlaps a slow tick | mitigate | The fired tick is idempotent via the 03-03 `lastTickDay` guard — a duplicate sweep cannot double-freeze |
| T-03-11 | Denial of Service | unbounded subs:registry / per-sweep cost | accept (this plan) | Registry is a bounded SET of installed communities (sAdd idempotent); per-hour enumeration is O(installed subs) — acceptable at hackathon scale; revisit if the install base grows |
| T-03-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages (RESEARCH Package Legitimacy Audit clean — native `Intl`, no library); slopcheck N/A |
</threat_model>
