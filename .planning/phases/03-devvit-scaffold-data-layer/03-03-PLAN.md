<objective>
Turn a day's accumulated Redis activity (from plan 03-02) into a frozen Ring record the universe can render. Define the Ring record contract (DayVector + seed + genomeVersion, ~25 scalars, no images — DEV-05), the Redis read/write service indexed by an explicit `ringCount` (DEV-05, no scan), and the scheduler `tick` handler that freezes the frontier: read accumulators → compose `conflict` → build+parse a Ring record with `seed = hash(subId, day, genomeVersion)` → write it → reset the day counters → open the next frontier, all guarded idempotently (DEV-04 freeze path).

Purpose: DEV-04 requires the tick to freeze the frontier and write a Ring record; DEV-05 requires those records to be indexed by an explicit `ringCount` with only ~25 scalars + seed (no scan, no images). This is the vertical slice that makes "a day froze" real: after this plan, invoking the tick for a community turns its accumulated counters into a stored, render-shaped Ring at `organism:{sub}:ring:{n}` with `ringCount` advanced and the day counters reset. The hourly sweeper that *fires* this tick at local midnight + the install settings it reads land in plan 03-04; the client that *reads* these rings lands in plan 03-05.
Output: A `RingRecordSchema`, a `ring.ts` read/write service, a `tick.ts` freeze service with an idempotency guard, a `tickJob.ts` boundary schema, a `scheduler.ts` router with the `/tick` handler, the `index.ts` mount, and the `devvit.json` `scheduler.tasks.tick` declaration.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Devvit scheduler → /tick handler | The scheduler `data` payload (`{ subId, day }`) is an untrusted boundary crossing into the server |
| accumulated proxies → Ring record | Spam-inflated counters must not poison the frozen ring's conflict/scalars |
| at-least-once scheduler delivery → Redis ringCount | A double-fire must not double-freeze or corrupt the explicit ringCount index |
| Redis hash → engine render | A malformed/partial stored ring must be rejected at the read boundary, never rendered raw |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-06 | Tampering | scheduler `data` payload → runTick | mitigate | `TickJobSchema.parse(data)` at the /tick boundary before runTick; reject malformed (V5) |
| T-03-07 | Tampering (data integrity) | tick double-fire (at-least-once) → ringCount | mitigate | Idempotent `runTick` guarded by stored `organism:{sub}:lastTickDay`; a re-fire for the same/earlier day is a no-op (OQ3) — at most one ring per local day |
| T-03-04 | Tampering | spam-inflated counters → frozen conflict | mitigate | Conflict is composed via the volume-normalized, saturating `conflictComposite` (03-02); contributors via SET cardinality — a single spammer cannot spike the frozen ring |
| T-03-05 | Denial of Service | unbounded per-day SET/ZSET growth | mitigate | runTick deletes/expires the day-scoped counters/SET/ZSET after the freeze consumes them (no unbounded accumulation across days) |
| T-03-08 | Tampering | malformed/partial stored ring → render | mitigate | `RingRecordSchema.parse` is the single read boundary in `readAllRings`; a partial hash is rejected, never trusted raw (V5, Pitfall 6) |
| T-03-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages (RESEARCH Package Legitimacy Audit clean — all Devvit/Hono first-party, pre-existing); slopcheck N/A |
</threat_model>
